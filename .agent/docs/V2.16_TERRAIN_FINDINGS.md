# V2.16 — Terrain Findings and Resolved Defects

## Scope

This document records empirical defects discovered while copying the V2.16 `.agent` bundle into isolated projects and then replaying the workflow on a realistic codebase.

It distinguishes terrain evidence from CI coverage. It does not mark the PR ready for merge and does not replace the remaining host-visible MCP gate.

## Real Graphify schema

The installed Graphify build completed with two nodes and one relation. Its `graph.json` used NetworkX node-link data with root fields:

```text
directed
multigraph
graph
nodes
links
hyperedges
```

The original readiness fixture assumed only `nodes + edges`; terrain therefore exposed a false `GRAPHIFY_CORRUPT` verdict after a successful build.

The readiness adapter now accepts only explicit supported edge collections:

```text
nodes + edges
nodes + links
```

It rejects absent collections, non-list collections and contradictory dual collections. `GRAPHIFY_READY.json` records the bound edge field.

## Concurrency defects exposed by expanded validation

### Partial TENOR lock creation

A waiter could read a lock while its owner was still writing JSON. An empty parse was interpreted as epoch zero, making a fresh lock appear stale. Age now falls back to the lock file `mtime`, while stale removal still rechecks ownership before unlinking.

### Windows manifest temporary-file collision

Concurrent writers could generate the same timestamp-derived temporary name. Manifest writes now use an exclusively created `tempfile.mkstemp()` file in the destination directory before `fsync` and atomic replacement.

### Host-instruction temporary-file collision

The host auto-repair writer still used `time.time_ns()` for `AGENTS.md` temporaries after the manifest writer had been hardened. It now also uses `tempfile.mkstemp()` and preserves atomic replacement. A concurrent test runs 64 repairs through 16 workers and requires one managed block with no temporary residue.

### Destructive Windows PID probe

`os.kill(pid, 0)` is not a POSIX-style existence probe on Windows. TENOR now uses read-only `OpenProcess` and `GetExitCodeProcess`, closes every handle and treats access denial as an existing but uninspectable owner.

### In-process finalization transaction

Concurrent finalizers could write `ready` and then observe another write between the write and gate inspection. Finalization now protects write plus inspection with a reentrant in-process transaction lock; the TENOR file lock remains the inter-process authority.

## Smoke fixture isolation

A Graphify smoke fixture once survived `mcp_smoke` and was then correctly rejected by the following red-team as `GRAPHIFY_FIXTURE_FORBIDDEN`.

Fixture lifecycle is now scoped:

1. snapshot prior canonical Graphify output;
2. install and authorize one fixture;
3. execute exactly one smoke subprocess;
4. restore prior output byte-for-byte or restore absence;
5. restore the previous environment even when the smoke fails.

Diagnostic logs live outside the checkout so Git hygiene remains strict.

## Raw-copy replay on `algowebsite` — host-binding defects

A later raw copy of the complete current `.agent/` into `algowebsite` proved that relocation and stale-Graphify refusal worked, but exposed four defects after local recovery:

1. the printed fallback used bare `python` on a POSIX machine where only `python3` existed;
2. TENOR did not mechanically detect/configure the actual project-local host;
3. the full proof bearer token appeared in terminal output and a manually piped JSON-RPC call;
4. a shell-launched MCP server could be presented in prose as `TENOR_INIT_READY` even though OpenCode had not exposed the tools.

The corrected contract now returns interpreter argv, manages only verified project-local OpenCode/Claude/Codex configuration, records a config-hash binding receipt, requires restart/reconnect after changes, rejects an ordinary shell process as `HOST_MCP_UNBOUND`, and consumes a server-side one-time proof without printing or persisting the full token. The internal `TENOR_INIT_BRIDGE_OK` receipt is promoted by the public wrapper to terminal `TENOR_INIT_READY` only after host-process, root and session proofs all pass.

Executable regression coverage now includes complete raw copy, destination memory present/absent, stale graph refusal, ambiguous host refusal, OpenCode JSONC preservation/idempotence, shell bridge rejection, current-interpreter actions, six concurrent proof issuers and single-use consumption. These are code/test proofs; the final OpenCode root-binding, micro-write and six-terminal terrain replays remain mandatory before merge.

## Real OpenCode TENOR TASK replay — governance bypasses

The first real host-bound task produced new evidence that invalidated the
earlier CI closure claim:

- descriptive prose was persisted as `intent`, then compared literally by the
  write gate, producing repeated `TASK_CONTEXT_INTENT_MISMATCH`;
- the model retired identities and registered replacements to escape HARD_STOP;
- one logical task created several active task contexts;
- three native OpenCode edits changed product files with zero MCP patch receipt;
- a `FIXED` SCRIBE record was staged without an applied MCP patch;
- the model ran global `graphify update .`, creating root `graphify-out/` and
  dirtying tracked/generated product paths;
- `test_multi_agent_resource_lock.py` was absent from CI and, when invoked
  directly, had no `unittest.main()`. Its old fixture was stopped by the TENOR
  installation gate before it could measure lock atomicity.

The corrected source uses a strict `read|write|delete` intent enum, one active
task per identity, retirement blockers, host-bound registration authority,
receipt-bound bugfix records, sequential multi-file rescoping and OpenCode
`edit/bash=deny`. Graphify now builds in an isolated mirror and is published
only to `.agent/state/outputs/graphify-out/`; a bound host uses
`graphify_project_build`.

The repaired resource-lock suite executes 23 real tests, including repeated
two-agent races. This proves lock atomicity; it does not yet replace the final
six-terminal `algowebsite` replay. PR nº 1 remains Draft and merge into `v2`
remains forbidden.

## Compact public task API — implementation evidence, terrain still required

The later OpenCode replay proved that exposing the fine-grained protocol to a
small model remained operationally unstable even after individual gates were
correct: one product fix consumed dozens of calls, repeatedly mismatched
intent/resource context, created replacement tasks/agents and finally bypassed
the MCP write path.

The source now advertises four normal task tools. `tenor_task_start` performs
task creation plus targeted SCRIBE/Graphify retrieval server-side.
`tenor_apply_changeset` owns multi-file path/scope/hash/lock preflight,
deterministic locking, durable staging, atomic commit/rollback, bounded
no-shell validators, runtime evidence and closure. `tenor_activity` consolidates
presence/current/last/next state. `tenor_task_control` is owner-only.

Local executable coverage proves normal two-file commit, stale-hash zero-write,
validator rollback, injected mid-commit rollback, idempotent retry, request-id
conflict, create/confirmed-delete, traversal/symlink refusal, process-bound
identity, two-agent ownership isolation, live/dead PID handling and rolling
task TTL. These remain code/test proofs. A new real OpenCode atomic commit,
rollback and six-terminal replay is still mandatory before merge into `v2`.
